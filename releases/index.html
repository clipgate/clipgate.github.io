Artifact Layout

Each release folder is designed to hold platform archives plus matching checksum files. The same structure is used by the installer and manual downloads.

macOS

Apple Silicon and Intel archives.

/releases/vX.Y.Z/cg-aarch64-apple-darwin.tar.gz /releases/vX.Y.Z/cg-x86_64-apple-darwin.tar.gz

Linux

x86_64 and ARM64 tarballs.

/releases/vX.Y.Z/cg-x86_64-unknown-linux-gnu.tar.gz /releases/vX.Y.Z/cg-aarch64-unknown-linux-gnu.tar.gz

Windows

MSVC zip archive for manual download.

/releases/vX.Y.Z/cg-x86_64-pc-windows-msvc.zip

How Publishing Works

The release pipeline is intentionally simple: build privately, publish publicly.

1. Build

Tag a version in the private CLI repo and compile the supported platform targets.

2. Package

Create tar.gz or zip archives and generate checksum files for each artifact.

3. Publish

Copy files into the public site repo under /releases/<tag>/, update latest.txt, publish the Python package to PyPI, and rewrite the Homebrew tap formula.

4. Serve

GitHub Pages exposes the binaries, docs, and release metadata while PyPI and the public Homebrew tap stay aligned with the same tagged release.
